什么是链接 What is Linking?

链接器的基本工作 What do Linkers do?

以一个简单的静态链接为例：

  • 输入：

    • 若干个.o文件。即多个可重定位目标文件（relocatable object file）；
    • 命令行参数：例如-Og、-o

  • 输出：一个a.out文件。即可执行目标文件（executable object file）。

  • 工作步骤：

    1. 符号解析 Symbol Resolution
    2. 重定位 Relocation

链接的意义 Why Linkers?

模块化 Modularity

  • 多个分离模块组成一个大的程序。程序可以按照功能分散成更小的源文件模块，实现解耦；

  • 可以将常用的函数/功能编写为库文件（library）。例如数学运算库，C标准库等。

高效率 Efficiency

  • 时间角度：支持分离式编译的重要支撑。某一个源文件的改变无需整个程序重新编译，单独编译修改了的源文件并重新链接即可；

  • 空间角度：常见的功能函数可以整合到一个模块/一个库（library）中，而可执行文件和运行的程序却只包含了使用的那部分功能函数的代码。

链接的时刻 When to link?

根据链接的不同时刻，可以分为：

编译时 Compile Time

即静态链接。

加载时 Load Time

加载器将可执行文件送入内存准备执行的过程中进行链接。属于动态链接的一种。

运行时 Run Time

进程执行的时候显式调用某些内核函数来实现运行时链接。属于动态链接的一种。

目标文件 Object File

Object files are merely collections of blocks of bytes (字节块).

三种形式 Forms

可重定位目标文件 Relocatable Object File (.o file)

  • 内容为二进制代码（code）和数据节（data section）组成；

  • Each .o file is produced from exactly one source (.c) file.

  • 其形式可以在编译时与其他可重定位目标文件合并，来创建出一个可执行目标文件。

可执行目标文件 Executable Object File (a.out file)

Contains code and data in a form that can be copied directly into memory and then executed.

不同系统下的可执行目标文件格式：

  • Linux: ELF.

    The first Unix systems from Bell Labs used the a.out format. (To this day, executables are still referred to as a.out files.)

  • Windows: Portable Executable, PE

  • Mac: Mach-O

共享目标文件 Shared Object File (.so file)

特殊类型的可执行目标文件，可以在加载（load）或者运行时（run）被动态加载进内存并链接。Called Dynamic Link Libraries (DLLs) by Windows.

可执行文件（a.out）的格式：

  • ELF header: Word size, byte ordering, file type (.o, exec, .so), machine type, etc.

    includes the file offset of the section header table, the size and number of entries in the section header table.

  • Segment header table (required for executables): Page size, virtual addresses memory segments (sections), segment sizes.

  • .text: The machine code of the compiled program 代码段

  • .rodata: read-only data 只读数据区，例如printf语句中的格式串以及switch语句中的jump tables

  • .bss: Uninitialized global and static variables, along with any global or static variables that are initialized to zero (未显式初始化，默认置0的情况).

    从空间效率角度出发，只是一个占位符（placeholder），不占用实际空间（Better Save Space）。

    术语.bss来形容未初始化的数据是很普遍的。始于IBM704汇编语言中“Block Storage Start”指令的首字母缩写，并沿用至今。

  • .symtab: Symbol table 符号表，其内容即为所有符号的集合，具体形式见后。

    每个.o文件中都在此节有一个符号表，除非是用STRIP命令显式去掉。

  • .rel.text: Relocation info for .text section

  • .rel.data: Relocation info for .data section

  • .debug: Info for symbolic debugging

  • .line: C source code 中的行号 与 .text节中机器指令 之间的映射。

    只有以-g选项调用编译器驱动程序时，才会得到这张表。

  • .strtab: 字符串表（string table），以 null 结尾的字符串序列。

    内容存放了一系列静态的字符串，.symtab、.debug以及 section header 中的 section name 都会指向这里的字符串。

  • Section header table: Offsets and sizes of each section.

    Contains a fixed-size entry for each section in the object file.

符号解析 Symbol Resolution

符号 Symbol

目标文件定义和引用符号。符号即一个个体，对应于一个特定的函数、全局变量（global）、或者是静态变量（static）。

一个符号定义与引用的实例：

全局符号 Global symbols

  • defined by module $m$. 由模块 $m$ 定义

  • can be referenced by other modules. 且能够被其他模块引用

例如：非静态（non-static）的函数与全局变量。

外部符号 External symbols

  • Global symbols that are: 一种特殊的全局符号

  • defined by some other module. 由其他模块定义

  • referenced by module m. 且被模块 $m$ 调用

即外部定义的函数或者变量（例如extern）在当前文件中引用。

本地符号 Local symbols

Defined and referenced exclusively by module m.
由模块本身定义，且对外部模块不可见。例如使用static来隐藏函数或者变量名。

区分 local symbols & local variable

符号解析是链接时期的工作，关注的符号都是存储在内存全局区的内容（只关注全局/静态变量以及函数），而过程中的局部变量则由编译器在栈上组织。

  • 非静态局部变量（local non-static variables）：处于栈中。这些符号由编译器来解析，而非链接器的工作内容；

  • 静态局部变量（local static C variable）：属于 local symbol. 编译器会在.data/.bss为其分配空间，并在 symbol table 中创建一个唯一名字的 local linker symbol.

符号表 Symbol Table

存储位置

Contained in the .symtab section.

构建

Built by assemblers, using symbols exported by the compiler into the assembly-language .s file.

  • 编译器在.s汇编文件中构建符号；

  • 汇编器根据这些符号构建符号表。

内容

  • Symbol table is an array of structs.

  • Each entry includes name, size and location of symbol and more:

条目内容

  • name: 符号名均存储在ELF中的.strtab字符串表中，这里的name是指向其位置的指针/字节偏移量。

  • section: 指明符号在目标文件的哪个节（section）。

    其中该字段的值可能指向三个特殊的伪节（pseudo-section），其在节头部表（Section Header Table）中没有条目。

    1. ABS: 不需要被重定位的符号；

    2. UNDEF: 外部符号，即外部模块定义，当前模块仅引用的符号

    3. COMMON: 未被分配位置、未初始化的（全局）数据目标；

      • COMMON: Uninitialized global variables
      • .bss: Uninitialized static variables, and global or static variables that are initialized to zero

      对于COMMON中的符号，value字段代表对齐要求，size则代表最小的大小。

    只有.o文件中存在伪节，exec文件中没有。

  • value: 结合section字段，指明符号的地址；

    • 对于.o文件：值为相对于section起始的偏移量；
    • 对于exec文件：值为绝对地址（无需重定位，直接可以执行），无需借助section字段来定位。

  • size: 指明大小，单位为字节。

READELF

GNU READELF是一个常用的查看目标文件内容的工具，如下为其输出。开始的8个条目都是连接器内部使用的局部符号，没有给出：

  • main为.text（根据Ndx=1确定）节偏移量为0处的全局函数符号，大小为24Bytes；

  • array为.data（根据Ndx=3确定）节偏移量为0处的全局数据对象符号，大小为8Bytes；

  • sum为未定义的外部符号，在外部定义，在该模块仅仅引用。

符号解析 Symbol Resolution

符号首先要被定义，其后会出现若干次引用。符号解析的过程是将 每个符号引用 与 符号表 中一个确定的 符号定义 关联起来 （associates each symbol reference with exactly one symbol definition.）。、

本地符号 Local Symbol

对于本地符号：引用和定义都在同一个模块。解析过程比较简单：

  • 不同模块的同名静态变量也会由编译器赋予唯一的名字。

  • 同一模块中的本地符号定义只能有一次，否则错误。

全局符号 Global Symbol

对于全局符号，符号的引用和定义很可能不在一个模块中（external），会出现一些棘手的情况：

  • 全局符号不在当前模块中定义，且无法在其他任何输入模块中找到；

  • 全局符号在多个模块中出现同名。

编译过程中，编译器向汇编器输出每个全局符号，并将其分为两种：强（strong）或弱（weak）。汇编器会将这个信息隐含的编码在.o文件的符号表中。

  • Strong: procedures and initialized global variables；

  • Weak: uninitialized global variables;

全局符号的定义缺失 Lack of Definition

编译器遇到一个符号引用，且该符号定义不在本模块中时，就会假设该符号在其他某个输入模块中定义。编译器为其生成链接器符号表的条目，交给链接器来处理。这个过程编译器会没有任何障碍的继续运行。

链接器如果在任何输入模块中都无法找到该被引用符号的定义，就会输出错误信息（undefined reference）并终止。

全局符号的多重定义 Duplicate Symbol Definition

Linux根据以下规则来解决多重定义的问题：

  1. 不允许出现多个强符号；

  2. 一个强符号与若干个弱符号的情况下，选择强符号；

  3. 若干个弱符号的情况下，随机选择。

    这些情况下，链接器允许同时存在多个同名全局符号。而编译器在遇到一个弱全局符号时是无法判断链接器会使用哪一个，所以需要将弱全局符号（即uninitialized global var.）送入COMMON区，这意味着由链接器来决定最终使用多重定义中的哪一个。相反的，对于静态或已经初始化的全局变量，编译器可以放心地送入.bss或.data区。

总结下来就是：强至多一个，有强选强，没强随机。

多重定义虽然被允许，但很有可能引发各种意想不到的错误或覆盖：

  • 多个弱符号可能带来歧义，引发一些不易察觉的运行时错误（不同模块的全局变量并不想共享）。链接器通常也不会提醒检测到多个同名定义；

  • 同名符号但不同类型，甚至类型大小不同。可能会存在错误覆盖的现象（overwrite）。

  • Nightmare scenario: two identical weak structs, compiled by different compilers with different alignment rules.

如果怀疑有这类错误：

  • GCC-fno-common：该标志会告诉链接器，在遇到多重定义的全局符号时，触发一个错误；

  • -Werror：把所有的警告编程错误。

重载与重整 Overload & Mangling

C++与Java等高级语言都支持方法重载，这意味着会有相同的名字/标识符与不同的参数列表。编译器会在编译过程中使用名字+参数列表的方式，来保证生成符号的唯一性，从而支持高级语言中的重载方法。这个编码的过程就被称为重整（mangling），相反的过程就称为 demangling.

从符号解析的角度，在全局变量（global variable）方面的编程建议：

  • Avoid if you can (use static to protect your data);
  • Initialize if you define a global variable;
  • Use extern if you reference an external global variable;

重定位 Relocation

  • Merges separate code and data sections into single sections

  • Relocates symbols from their relative locations in the .o files to their final absolute memory locations in the executable.

  • Updates all references to these symbols to reflect their new positions.

静态库 Static Libraries

对于非常常用的一些系统函数（基本数学操作、I/O操作等），我们希望将其打包以供使用。可以有以下几种实现方案：

  1. 编译器负责辨认标准函数的调用并负责生成响应代码。对于程序员来说比较方便。

    缺点：编译器增加十分显著的复杂性。编译器版本需要随着标准函数的修改而进行频繁迭代。

  2. 所有标准函数放在一个单独的可重定位目标模块中。将编译器的实现与函数的实现分离开来。

    缺点：空间大幅浪费。且重新编译需要耗费巨大（时间）资源。

  3. 每个标准函数都声称对应的单个独立模块。

    缺点：程序员需要显式链接出所有需要的接口，易错且耗时。

根据以上方案的分析，可以进行折中处理：及那个相关的函数们编译成独立的目标模块（.o），并将这些模块封装成一个单独的静态库文件。

libc.a与libm.a这类静态链接库也属于比较早期的实现。曾经这些C标准库在Linux每个发行版本中都会有，但现在新的发行版中都已经被glibc而取代。

什么是静态库 What are Static Libraries?

  • 一系列可重定位目标模块的集合（使用索引组织）；

  • 格式为.a，称为 archive 文件；

  • 链接时，如果 archive 中的某一成员模块能够解析引用，则将其链接至可执行文件的一部分。

    即，只会复制 archive 中被程序引用的目标模块。

与静态库链接 Linking with Static Libraries

一个实例过程如下：

  • -static：告诉编译器驱动程序，链接器需构建一个完全链接（fully-linked）的可执行目标文件（executable）。其可以直接加载到内存并运行，在加载时无需更进一步的链接。

  • -L.；在当前目录（.）下查找；

  • -lvector：缩写，等价于libvector.a。

链接器判断出 archive 中引用了哪些模块，并将其复制到可执行文件中，而不会复制任何未引用的模块。

使用静态库解析符号引用 Use Static Libraries to Resolve References

符号解析的过程中：链接器 从左到右，按照在命令行上出现的顺序 来扫描所有的输入。这意味着，命令行上的输入模块的顺序十分重要（Command line order matters!），且符号的定义一定要出现在符号的引用之后！

链接器在扫描过程中，维护如下集合：

  1. 可重定位目标.o文件集合 $E$；

  2. 引用但未定义/解析的符号集合 $U$；

  3. 已定义/解析的符号集合 $D$；

解析流程如下：

  1. 初始状态：$E, U, D$ 均初始化为空集；

  2. 对于每个输入文件/模块 $f$ ，判断其为 object 还是 archive；

  3. 如果是 object 文件，解析其中的符号，相应的送入集合 $E$ 与 $D$ ；

  4. 如果是 archive 文件，则尝试使用 archive 中的成员来解析 $U$ 中的符号。

    根据解析的结果，修改 $U$ 和 $D$，直至两个集合都不再变化。剩余的成员直接丢弃，继续处理下一个输入文件。

  5. 扫描完成后，如果 $U$ 非空，意味着有无法解析的引用，则报错（undefined reference）并终止。否则，合并集合 $E$ 中的目标文件，构建输出的可执行文件。

因为命令行的顺序非常重要，所以关于库的引用有一些准则：

  • 库通常放在命令行结尾；

  • 相互独立的库顺序不用关心，不是相互独立的库必须要进行排序；

  • archive 库在排序时保证至少有一个符号 $s$ 的定义要在其引用之后；

  • 如果存在依赖上的环路，可以在命令行上重复库。例如：foo.c 依赖于 libx.a，libx.a 调用 liby.a 中的函数，而 liby.a 也调用了 libx.a 中的函数。

    当然，更好的方式是封装archive时将这种情况都合并在一个单独的文件中。

动态链接Dynamic Linking

共享库 Shared Libraries

静态库是一种折中的实现，仍然不是最现代化最有效率的实现方式。其仍然会复制部分公共的函数到每一个程序的内存中，从而造成稀缺的内存资源浪费。

  • Duplication in the executables. Every function needs libc.

  • Minor bug fixes of system libraries require each application to explicitly relink.

共享库用于解决这个问题。

加载时 Load-time

Dynamic linking can occur when executable is first loaded and run.

  • Common case for Linux, handled automatically by the dynamic linker. (ld-linux.so)

  • Standard C lib (libc.so) usually dynamically linked.

运行时 Run-time

高级语言代码中调用dlopen。

In Linux, this is done by calls to the dlopen() interface. For:

  • Distributing software

  • High-performance web servers

  • Runtime library interpositioning

Shared library routines can be shared by multiple processes. (virtual memory)

位置无关代码 Position-Independent Code, PIC

  • 多个正在运行的进程能够共享内存中相同的动态库代码。

  • 共享模块的代码段可以加载到内存的任何位置而无需链接器修改；

  • 可以加载而无需重定位的代码就成为位置无关代码。

库打桩机制 Library Interpositioning

  • 截取对共享库函数的调用，取而代之执行自己的代码（来进行调试）；

  • 创建wrapper来欺骗系统调用wrapper而非target functions.
